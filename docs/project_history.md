# Sato Clone Project - アーキテクチャと開発履歴リファレンス

本書は、Sato Digital Clone Organization（佐藤デジタルクローン組織）の頭脳コア部分を構築・拡張した一連の開発プロセス、アーキテクチャ、および使用技術に関する詳細な記録です。IT知識の向上と将来的な保守・拡張のためのリファレンスとして機能します。

---

## 1. プロジェクトの大目的と背景
**大目的:** 
単なる「プロンプトに対してWeb検索をして返すAI」ではなく、ユーザー（佐藤氏）の関心ごとを自ら記憶し、学び、**主体的な思考コア（クローン）として振る舞うシステム**を構築すること。

**細かな要望:**
1. LLMによる都度のWeb検索（コストと時間を食う）を減らし、**蓄積された知識（脳）を優先**して回答する。
2. 過去の会話の文脈を理解し、フォローアップの質問に耐えられるようにする。
3. クローン自身が「Googleの汎用LLM」ではなく「佐藤クローンの思考コア」であるという確固たる**自己認識**を持つ。
4. ユーザーが放置していても、クローンが自律的にインターネットを巡回し、最新のAIニュースや興味あるトピックを**自動で学習・記憶**する。
5. 「今調べて」などの明確な指示があった場合は、機敏にWeb検索を行いながらそれを知識として取り込む。

---

## 2. 実装された新アーキテクチャ（脳ファースト）

本プロジェクトで構築されたハイブリッド・マルチエージェント・パイプラインのフローは以下の通りです。

1. **ユーザー入力受付**
   - 意図分類：`orchestrator.py`の正規表現により「雑談か」「今調べてか」「深い考察か」を判別。
2. **記憶の検索（SQLite `brain.db`）**
   - ユーザーの質問に関連する情報を、過去の記憶や事前知識からベクトルライク（本番はLIKE検索）に取得。
3. **自律ハイブリッド化**
   - 記憶が豊富なら、そのまま手順4へ（ゼロコスト・爆速化）。
   - 記憶がなければ、裏で一時的に`Research`エージェントを呼んでWeb検索し、その結果を直ちに「脳に保存（学習）」してから手順4へ。
4. **思考と回答生成**
   - 直近数ターンの**会話履歴**、検索した**脳の記憶一覧**、そして**「自分は佐藤クローンである」という強いシステムプロンプト**をすべて繋ぎ合わせ、`Planning`エージェントが一つの知的な回答を組み立てる。

## 3. 自律パトロール機能（クローラー）の詳細

自動学習を担う `core/patrol.py` は以下の特徴を持つ自律ワーカーです。

- **トリガー**: `infrastructure/start_patrol.sh` によりバックグラウンドで常駐し、APSchedulerによって定期的（デフォルト6時間ごと）に起動。
- **情報源の統合**:
  1. `config/patrol_config.yaml` の静的トピック
  2. `brain.db` の `interests` テーブル（ユーザーが過去に興味を示した質問キーワード）
  3. `data/patrol_sites.txt`（ユーザーが自由に編集できる巡回先URL・キーワードリスト）
- **重複防止・コスト削減**: 直近3日間に同じトピックが脳に記憶されている場合はスキップ。
- **レジューム機能**: どこまで調べたかを `data/patrol_state.json` に記憶し、再起動しても途中から再開可能。

## 4. 利用技術・ミドルウェア・制限事項の詳細

### 4.1. バックエンド・データベース
- **Python 3.x**: オーケストレーター(`orchestrator.py`)、パトロール(`patrol.py`)、DB操作(`brain.py`)などすべての制御を担当。
- **SQLite3**: クローンの「脳」。RDBMS（リレーショナルデータベース）。
  - **WAL（Write-Ahead Logging）モード**: 有効化済み。高速な読み書きと、複数プロセスからの同時アクセスによるロック競合を回避。
  - 特徴: `TEXT`型に大量の文字を圧縮して格納し、`text_factory` で不正なサロゲート文字を吸収してエラーを防ぐ堅牢な実装。

### 4.2. PicoClaw（マルチエージェント・フレームワーク）
Sipeed社が提供する軽量マルチエージェント・オーケストレーションシステム。
- **Topology (トポロジー)**: `config/topology.yaml` にて、組織内の役割（総務部、企画部、調査部など）を定義。それぞれが独立したシステムプロンプト(`IDENTITY.md`)を持つ。
- **Pub/Sub通信**: エージェント同士はMQTT（メッセージキュー）を通じて非同期に情報をやり取りする。
- **Heartbeat**: 生存確認のための機能。30分に1回、PicoClaw内部で相互にPingを飛ばし合っている（今回はPythonの自律パトロールで補完）。

### 4.3. Docker アーキテクチャ
- **`sato-mqtt-broker`**: コンポーネント間通信のハブとなるEclipse Mosquitto。
- **`sato-picoclaw-core`**: 複数のエージェントを動かす心臓部。ホスト（ubuntu等の実機）の `config` や `data` ディレクトリを**Volumeマウント**し、コンテナ内から読み書きできるようにしている。
- **パーミッションの壁**: Dockerコンテナ内（root権限）とホストPC（ユーザー権限）でファイルを作成し合うと `Permission Denied`エラーが起きやすい。今回はPython側でフォールバック機構（エラー時は別の場所に書き込む）を作って回避。

### 4.4. LLM (Google Gemini 2.5 Flash) の仕様と限界
- **使用モデル**: `gemini-2.5-flash`。非常に高速でコストパフォーマンスが高い。
- **コンテキストウィンドウ**: 約100万トークン。一冊の本丸ごとを記憶できる限界だが、API料金はトークン長に比例するため、必要な知識（数百文字）だけをSQLiteから引いて渡す「脳ファースト」設計がコスト的に極めて重要。
- **コスト感**: 1回のパトロール（3トピック検索+要約）で約 $0.002（約0.3円）。
- **課題**: `IDENTITY.md`（システムプロンプトのファイル指定）がPicoClawからGemini APIへ正しく伝達されないバグが存在。
  - **対応録**: Python (`orchestrator.py`) で毎回の質問プロンプトの頭に「あなたはGoogleのAIではなく、佐藤クローンです」と強制的に文字列結合させることで物理的に回避した。

## 5. トラブルシューティングと対応履歴

| 問題点・症状 | 原因 | 解決策（実装） |
|---|---|---|
| **「Googleの言語モデルです」と名乗る** | 外部のプロンプトファイル（IDENTITY.md）がLLMに無視される仕様だった | Python側で生成する最終プロンプトの冒頭に直接「あなたは佐藤クローンである」という指示文字列を埋め込んだ。 |
| **会話の文脈（前回の質問）を忘れる** | API呼び出しは毎回「初めての会話」として扱われるため（ステートレス） | `orchestrator.py`内に直近5ターンの会話履歴配列を持たせ、毎回プロンプトに「直近の会話：〇〇」とテキストで同梱して送信するようにした。 |
| **sqlite3.OperationalError: database is locked** | 複数プロセス（CLIとパトロール）が同時に書き込もうとした | DB接続オプションに `PRAGMA journal_mode=WAL;` を追加し、書き込み待ち時間を劇的に短縮した。 |
| **UnicodeEncodeError (surrogates not allowed)** | Web上の不正な文字コード（絵文字の断片など）をDBが拒否した | `_sanitize` 関数を作成して不正文字を削り、DBの読み込み設定でエラー文字を `replace` するようにした。 |
| **「情報が不足しています」という回答がそのまま脳に保存される** | LLMがWeb検索に失敗した際の苦し紛れの回答がそのまま学習されてしまった | `REJECT_PATTERNS` というNGワードリストを定義し、「Google」「大規模言語」などの単語が含まれる回答はSQLiteに `store()` しない品質ガードを設けた。 |
| **Permission Denied (ディレクトリへの書き込み不可)** | Dockerが生成したディレクトリに、ホスト側のPythonがアクセスしようとした | エラーを `try-except` で捕捉し、保存できなかった場合は一段階上のルートディレクトリにファイルを退避させる安全策を追加。 |
